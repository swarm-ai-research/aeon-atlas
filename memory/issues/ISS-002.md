# ISS-002 — atlas-layers silently skipping its weekly Sunday 05:00 dispatch

## Symptom

`atlas-layers` is `enabled: true` in `aeon.yml` with `schedule: "0 5 * * 0"` (Sunday 05:00 UTC). In `memory/cron-state.json` it is frozen at:
- `last_dispatch: "2026-07-05T05:08:16Z"`
- `last_success: "2026-07-05T05:13:02Z"`
- `last_status: "success"`, `consecutive_failures: 0`, `success_rate: 0.88`

As of 2026-07-19 09:13 UTC the skill has not dispatched for **14+ days** — 2× its weekly interval — missing two consecutive Sunday slots (07-12 and 07-19).

## Evidence it is a dispatch/scheduling problem, not a content failure

1. `last_status` is still `success` and `consecutive_failures` is `0` — the skill is not failing, it simply is **not being dispatched**. A run that failed would have flipped these.
2. The sibling weekly skill `atlas` (`schedule: "0 4 * * 0"`) **did** dispatch on 2026-07-19 (last_dispatch 04:33, last_success 04:39). So the Sunday cron path is alive for the 04:00 slot but not the 05:00 atlas-layers slot.
3. This was tracked as a soft "slot-miss / escalation standing" in heartbeat logs from 07-12 onward; the pre-registered escalation ("if atlas-layers has not dispatched by 07-19 05:00, file an issue") is now met.

## Impact

- `quartz/content/atlas-layers.md` (seven-layer categorical view) has not refreshed since 2026-07-05 — the public layer view is going stale.
- No fleet-health alarm fires from the metrics themselves (last_status stays `success`), so the miss is silent except to heartbeat's P3 schedule-staleness check.

## Suggested next step (for skill-repair / config-validator)

Compare the workflow cron matcher's handling of the `0 5 * * 0` slot against the working `0 4 * * 0` slot. Check whether atlas-layers' entry is being matched by the scheduler at all on Sundays (e.g. a slot-collision, a matcher that only processes the first Sunday skill, or a run-frequency guard suppressing it). Confirm against the next Sunday (2026-07-26) dispatch, then reset expectations once it fires cleanly.
